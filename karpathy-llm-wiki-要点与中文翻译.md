## 一、核心要点（Key Takeaways）

1. **核心范式转变：从“查询时临时检索”到“持续编译知识”**  
   传统 RAG 在每次提问时都要重新从原始文档找片段并拼接。LLM Wiki 的思路是：让 LLM 持续维护一个持久化、可交叉链接的 wiki，把知识“编译”到中间层。

2. **三层架构清晰解耦**  
   - `Raw sources`：原始资料，只读、不改。  
   - `Wiki`：LLM 维护的结构化 Markdown 知识层。  
   - `Schema`：行为与结构规范（如 CLAUDE.md / AGENTS.md），约束 LLM 如何摄取、问答、维护。

3. **三大操作闭环：Ingest / Query / Lint**  
   - **Ingest**：新资料进入后，LLM 不只做索引，而是更新多页面、处理冲突、补充交叉引用。  
   - **Query**：基于 wiki 回答问题，并可把高价值回答回写为新页面。  
   - **Lint**：定期体检 wiki（冲突、过期信息、孤儿页、缺失概念、可补资料）。

4. **index.md 与 log.md 是可扩展关键**  
   - `index.md`：内容导航（按主题组织页面+简介）。  
   - `log.md`：时间流水（摄取/查询/lint 事件）。  
   二者让系统在中等规模下无需复杂向量基础设施，也能高效运作。

5. **工具是可选模块，模式才是核心**  
   Obsidian、qmd、Dataview、Marp、Git 都是增强件；真正核心是“LLM 作为 wiki 维护者，人类负责选材与判断”。

6. **为什么可行**  
   维护知识库最难的是“枯燥的持续整理”，不是思考本身。LLM 正好可低成本完成高频维护动作，使知识库长期保持活性。

## 二、中文翻译（精要版）

> 说明：以下为结构化精译，保留原文思想与逻辑，不做逐句全量直译。

### 1）什么是 LLM Wiki
LLM Wiki 是一种用大模型构建个人/团队知识库的模式。与常见“上传文档→问答时检索”不同，它让模型持续维护一套结构化 wiki。每次加入新资料时，模型会把新信息整合进已有页面，更新摘要、补链接、标注冲突，而不是每次提问都从零拼装答案。

### 2）核心差异
传统 RAG 的知识不会自然累积；复杂问题需要反复检索和重组。LLM Wiki 把“累积后的知识状态”沉淀到可读可改的 markdown 页面中：链接关系提前建立，冲突提前显化，综合结论持续演进。

### 3）人机分工
人负责选择资料、提出问题、做判断；LLM 负责摘要、归档、交叉引用、页面维护。可以把 Obsidian 看成 IDE，把 LLM 看成程序员，把 wiki 看成代码库。

### 4）适用场景
可用于个人成长追踪、主题研究、读书伴随知识图谱、团队内部知识库、竞品分析、尽调、课程学习等所有“长期积累型”任务。

### 5）推荐架构
- **原始层（Raw）**：论文、文章、图片、数据等原始证据，保持不可变。  
- **知识层（Wiki）**：由模型维护的实体页、概念页、专题页、综述页。  
- **规则层（Schema）**：定义目录结构、命名规范、摄取流程、回答格式、维护策略。

### 6）操作流程
- **摄取（Ingest）**：新增资料后，模型读取并讨论要点，写摘要，更新索引，联动更新相关页面，记录日志。单次摄取可影响多个页面。  
- **查询（Query）**：优先在 wiki 中检索并综合回答，输出可为文档、表格、图表、幻灯片等；高价值回答可反写 wiki。  
- **体检（Lint）**：定期检查冲突、过期、孤儿页、缺失概念和可补证据点，持续提升知识库质量。

### 7）索引与日志
- `index.md` 用于“内容导览”，帮助快速定位相关页面。  
- `log.md` 用于“时间追踪”，记录每次摄取、问答和体检。统一标题前缀后可被 grep 等工具直接解析。

### 8）可选工具建议
小规模先用 index 即可；规模扩大后可加入本地检索工具（如混合检索）。网页剪藏、图片本地化、图谱视图、前言字段查询、Git 版本管理都能显著提升可维护性。

### 9）结论
这不是某个固定产品，而是一种可迁移的工作模式：先把“知识持续编译为 wiki”做对，再按场景替换工具与实现细节。

**LLM Wiki = 用 LLM 持续把多源原始资料“编译”为可追溯、可交叉链接、可长期演化的结构化知识库。**
